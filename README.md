Bank API
REST API для банковского сервиса на Go с поддержкой JWT-аутентификации, выпуском виртуальных карт (PGP шифрование), кредитованием и аналитикой.

О проекте
Bank API - учебный проект, демонстрирующий создание безопасного банковского сервиса на Go. Проект реализует полный цикл операций: от регистрации пользователей до кредитования с интеграцией внешних сервисов (ЦБ РФ, SMTP).

Цель проекта: закрепить навыки разработки REST API на Go, работы с PostgreSQL, реализации безопасности (JWT, PGP, bcrypt), интеграции с внешними API и написания тестов.

Особенности реализации
1. Проверка прав доступа
Все запросы к защищённым ресурсам проходят через middleware, который извлекает userID из JWT и кладёт в контекст. В репозиториях используется JOIN с user_id, что гарантирует, что пользователь не получит доступ к чужим счетам или картам даже при подмене ID в запросе.

2. Шифрование карт
Номер карты и срок действия шифруются PGP (RSA 2048) перед сохранением в БД. HMAC-SHA256 вычисляется от зашифрованного номера для проверки целостности. CVV хешируется через bcrypt и не хранится в открытом виде. При просмотре карт номер маскируется (первые 6 и последние 4 цифры видны).

3. Расчёт кредита
Используется аннуитетная формула платежа. Конечная ставка = ключевая ставка ЦБ РФ + 5% (маржа банка). При недоступности ЦБ используется fallback-значение 16% с логированием ошибки.

4. Шедулер
Запускается в отдельной горутине при старте приложения. Каждые 12 часов проверяет просроченные платежи по кредитам. При наличии средств на счету - списывает. При отсутствии - начисляет штраф 10% от суммы платежа. При просрочке более 90 дней кредит переводится в статус "defaulted".

5. Прогноз баланса
Учитывает текущий баланс счёта и вычитает все непогашенные платежи по кредитам пользователя на указанный период. Максимальный период - 365 дней.

Решение проблем и частые вопросы
Проблема: PGP ключ не задан, карты не шифруются
Решение: Сгенерируйте ключи командой go run scripts/gen_keys.go и добавьте PGP_PUBLIC_KEY в переменные окружения. Без ключа данные карт хранятся в base64, что небезопасно для production.

Проблема: При просмотре карт номер отображается некорректно
Решение: Убедитесь, что маскирование номера происходит только один раз - в сервисе при создании карты. Хендлер не должен повторно маскировать номер.

Проблема: Шедулер не завершается при остановке сервера
Решение: Добавьте defer scheduler.Stop() после запуска шедулера в main.go для graceful shutdown.

Проблема: Не приходят email уведомления
Решение: Для тестирования используйте MailHog (docker run -d -p 1025:1025 -p 8025:8025 mailhog/mailhog). Для реальных писем настройте SMTP через Mailgun или SendGrid.

Проблема: Ошибка при запросе к ЦБ РФ
Решение: Сервис имеет fallback-значение 16% при недоступности ЦБ. Ошибка логируется, но приложение продолжает работать.

Production-ready улучшения
Что необходимо добавить перед запуском в реальной эксплуатации.

1. Хранение секретов
Проблема: Сейчас все ключи (JWT_SECRET, HMAC_SECRET, PGP ключи) хранятся в переменных окружения или .env файле. Это небезопасно для production.

Решение: Использовать HashiCorp Vault или облачные аналоги:

AWS Secrets Manager

Google Cloud Secret Manager

Azure Key Vault

2. Ротация секретов
Проблема: Ключи никогда не меняются. При компрометации злоумышленник получает постоянный доступ.

Решение:

JWT_SECRET менять каждые 30-90 дней

Хранить историю последних 2-3 ключей для валидации старых токенов

HMAC_SECRET для карт менять раз в год (потребуется перешифровка всех карт)

3. Мониторинг и алертинг
Проблема: Сейчас нет метрик, непонятно, что происходит на сервере.

Решение:

Добавить метрики через Prometheus (количество запросов, ошибки, latency)

Визуализация через Grafana

Настроить алерты в Telegram/Slack при падении сервиса или превышении порогов ошибок

4. Graceful shutdown для шедулера
Проблема: При остановке сервера горутина шедулера остаётся висеть.

Решение: Уже описано в разделе "Решение проблем". Добавить:
defer scheduler.Stop()

5. Rate limiting
Проблема: Нет защиты от DDoS и брутфорса.

Решение: Использовать библиотеку golang.org/x/time/rate. Настроить лимиты:

10 запросов в секунду на один IP

5 попыток входа в минуту (защита от брутфорса)

100 запросов в секунду общий лимит

6. TLS/HTTPS
Проблема: Все запросы идут по HTTP, данные передаются в открытом виде.

Решение:

Получить SSL сертификат (Let's Encrypt бесплатно)

Настроить HTTPS с HTTP/2

Добавить HSTS заголовки

Использовать строгие TLS настройки (TLS 1.2 и выше)

7. Логирование в production
Проблема: Логи пишутся в stdout и теряются при перезапуске.

Решение:

Настроить агрегацию логов (ELK стек или Loki)

Уровень логирования выставить в INFO (не DEBUG)

Структурированное логирование в JSON формате для удобного парсинга

Отдельный алерт на ошибки уровня ERROR и FATAL

8. Балансировка и масштабирование
Проблема: Приложение запущено в одном экземпляре.

Решение:

Добавить поддержку горизонтального масштабирования

Вынести сессии/JWT в Redis (хотя JWT самодостаточен, но для отзыва токенов нужен Redis)

Шедулер должен работать только на одном экземпляре (использовать Redis локи или отдельный воркер)

9. Graceful shutdown полный
Проблема: При остановке сервера могут потеряться запросы.

Решение:

Увеличить таймаут shutdown до 30 секунд

Добавить обработку сигналов SIGINT, SIGTERM, SIGHUP

Дождаться завершения всех обработчиков перед выходом

10. Аудит безопасности
Проблема: Нет логирования критических операций.

Решение:

Логировать все попытки входа (успешные и нет)

Логировать операции с картами (выпуск, оплата)

Логировать изменения баланса

Сохранять IP адрес и User-Agent для каждого запроса

11. Бэкапы базы данных
Проблема: При сбое данные могут быть потеряны.

Решение:

Настроить автоматические ежедневные бэкапы PostgreSQL

Хранить бэкапы в отдельном надежном месте (S3, отдельный диск)

Регулярно тестировать восстановление из бэкапа

12. Health check endpoints
Проблема: Оркестратор (Kubernetes, Docker) не может проверить работоспособность сервиса.

Решение:

Добавить GET /health - проверка что сервис жив

Добавить GET /ready - проверка что сервис готов принимать трафик (БД подключена, внешние сервисы доступны)
